rbk::Auth — generic session / login engine

Cookie based session authentication for Boost.Beast servers built on rbk. Header: rbk/HTTP/authHandler.h · Requires RBK_WITH_BOOST_BEAST (CMake) or WITH_BOOST_BEAST (qmake).

Design in one paragraph

The engine owns the mechanics: whitelist matching, the session cookie lifecycle (read / set / delete, TTL, secure flag), the redirect-vs-JSON-401 split for API calls, rendering the login form and the guard<> privilege wrapper for the routing table. It owns no user class and stores no user pointer: every user-related operation goes through the callbacks you install in rbk::Auth::Conf. Your project keeps its own user type and stores the pointer wherever it likes (thread local, request context, …). That is what makes the engine reusable across projects with completely different user models.

Request flow

request │ ▼ BeastConf::loginManager ──► rbk::Auth::loginManager │ ├─ path in publicPaths / publicPrefixes? ──────────────► serve, no auth │ ├─ session cookie present? │ │ │ └─► resumeSession(status, sessionId) │ ok ──► continue │ invalid ──► delete cookie, redirect login?error=7 (or JSON 401 for api/) │ notLogged ──► redirect login (or JSON 401 for api/) │ ├─ no cookie: develLogin(status) returned true? ───────► continue (development auto login) │ ├─ neither ────────────────────────────────────────────► redirect login (or JSON 401 for api/) │ └─► postLogin(status, payload) optional project checks (ACL, scoping, ...) │ ▼ routing table ──► guard<minLevel, handler> ├─ !isLogged() ──► 401 "login required" ├─ !hasLevel(minLevel) ──► 403 "insufficient privileges" (+ audit hook) └─ handler(status, payload)

Wiring, step by step

1. Fill rbk::Auth::conf() at startup

Call this once, before the server starts. Complete real-world example in digitalSpine/http/auth_handler.cpp (Auth::setup()).

#include "rbk/HTTP/authHandler.h"

void setupAuth() {
	using namespace rbk::Auth;
	auto& c = rbk::Auth::conf();

	//paths served WITHOUT a logged in user (leading / already stripped)
	c.publicPaths    = {{"login", true}, {"logout", true}, {"favicon.ico", true}};
	c.publicPrefixes = {"js/", "css/", "img/"}; //static assets, device traffic, ...

	c.cookieTTL     = 3600 * 24;          //seconds
	c.cookieSecure  = true;               //Secure flag on the cookie
	c.loginTemplate = "html/login.html"_p; //mustache, receives BASE_PATH + ERROR_MESSAGE
	c.successPath   = "index";             //landing page after login

	//---- the hooks: this is YOUR user model, the engine never sees it ----

	c.resumeSession = [](PMFCGI& status, const std::string& sessionId) {
		myCtx().user = MyUser::fromSession(sessionId);
		if (!myCtx().user)             return SessionState::invalid;   //cookie gets deleted
		if (!myCtx().user->isLogged()) return SessionState::notLogged; //back to the login page
		return SessionState::ok;
	};

	c.login = [](PMFCGI& status, const QString& email, const QString& password) -> LoginOutcome {
		switch (myCtx().user->login(email, password)) {
		case Err::ok:              return {LoginResult::ok, myCtx().user->registerSession()};
		case Err::invalidEmail:    return {LoginResult::invalidEmail, {}};
		case Err::invalidPassword: return {LoginResult::invalidPassword, {}};
		default:                   return {LoginResult::error, {}};
		}
	};

	c.logout = [](PMFCGI& status, const std::string& sessionId) {
		if (auto user = MyUser::fromSession(sessionId); user) user->logout();
	};

	c.isLogged = [] { return myCtx().user && myCtx().user->isLogged(); };
	c.hasLevel = [](int minLevel) { return myCtx().user->level >= minLevel; };
}

2. Register the engine in the beast config

setupAuth();
beast.conf.loginManager = rbk::Auth::loginManager;

auto& r = beast.conf.routingSimple;
r.insert({"login",  rbk::Auth::loginPage});
r.insert({"logout", rbk::Auth::logout});

3. Guard the routed handlers

guard takes the minimum level as an int so any project enum works; the comparison itself is your hasLevel callback. routingSimple stores plain function pointers, hence a template and not a lambda.

r.insert({"index",           rbk::Auth::guard<MyUser::basic, indexPage>});
r.insert({"api/admin/nuke",  rbk::Auth::guard<MyUser::admin, nukeApi>});

Tip: a project-local alias keeps the routing table readable with your own enum type (see Auth::guard in digitalSpine):

template <MyUser::Level minLevel, void (*handler)(PMFCGI&, Payload&)>
void guard(PMFCGI& s, Payload& p) {
	rbk::Auth::guard<static_cast<int>(minLevel), handler>(s, p);
}

Settings

FieldDefaultMeaning

publicPrefixesemptystartsWith matches — static assets, device traffic. Websocket upgrades happen before the engine is even called, so they never need whitelisting.

isApiCallstartsWith("api/")Paths for which a denied request gets a JSON 401 instead of a redirect to the login page.
loginErrorMessagebuilt inOverride the error=N → message mapping of the login page (see below).

Hooks

CallbackContract
resumeSessionrequiredResolve the session cookie to a user on your side (and set IP / UA there if you track them). Return ok, invalid (unknown session — the engine deletes the cookie) or notLogged (session resolved but the user is not, or no longer, logged).
loginrequiredCheck the credentials. On LoginResult::ok also register the session and return its id in LoginOutcome::sessionId — it becomes the cookie value.
logoutrequiredInvalidate the session server side. The engine deletes the cookie and redirects to the login page afterwards.
isLoggedrequired*Used by guard / denyRoute. *Only required if you use guard; an unset isLogged makes every guarded route answer 401.
hasLevelrequired*Used by guard: does the current user reach minLevel? The int is whatever your project enum casts to.
develLoginoptionalAuto login for development (e.g. credentials from the config file). Called only when no session cookie is present; return true if a user was logged in.
postLoginoptionalRuns after a successful preflight — ACL, account scoping, tenant checks. Return false to short circuit the request; in that case write the response yourself.
prepareAnonymousoptionalCalled at the start of loginPage, typically to set up an anonymous user object so logging / audit works before authentication.
auditBlockedRouteoptionalCalled by denyRoute on a 403, before the JSON error is written.

Login page error codes

The engine redirects to login?error=N and the login form renders ERROR_MESSAGE accordingly. Defaults (override with loginErrorMessage):

error=WhenDefault message
1LoginResult::invalidEmailInvalid email or username
2LoginResult::invalidPasswordInvalid password
3LoginResult::error (anything else)Login Error
7unknown / expired session cookieInvalid session

Denying from inside a handler

When a check deeper than the routing-table level fails, short circuit with:

rbk::Auth::denyRoute(status, payload, 401); //not logged  → {"status":"error","message":"login required"}
rbk::Auth::denyRoute(status, payload, 403); //no privilege → audit hook + "insufficient privileges"

rbk::Auth::writeJsonError(payload, code, msg) is also exported for generic JSON errors.

Notes & gotchas

  • Session id quality is on you. The cookie value is whatever login returns: use CSPRNG entropy (≥ 32 bytes, hex encoded), never std::rand.
  • Call conf() filling before listen(). The engine reads the singleton on every request but installing callbacks is not thread safe — it is startup configuration, not runtime state.
  • postLogin returning false must write the response — the engine writes nothing in that case.
  • The exception path of develLogin is yours too: throwing HttpException for a misconfigured devel user is fine and gives a clear error page.
  • Websocket upgrades bypass loginManager entirely (they are upgraded before it runs) — authenticate them separately if needed.
Reference implementation: digitalSpine/http/auth_handler.cpp shows the full wiring against a real user class (SpineUser): APCU-backed sessions, a thread-local request context (dk().user), development auto login from the config file, per-account scoping in postLogin and an audit trail on blocked routes.
